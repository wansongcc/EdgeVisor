# Agent Bench

This directory contains a minimal LangGraph-based agent episode runner for
comparing LLM generation backends while keeping tool execution and episode
control identical.

The runtime graph is:

```text
LLM generation -> tool call -> LLM generation
```

Backends currently implemented:

- `prima`: runs `/home/byh/B01/prima_cpp_work/prima.cpp/llama-cli`
- `edgevisor`: runs EdgeVisor `dllama` with root/worker processes
- `edgevisor_ablation`: runs EdgeVisor with ablation CLI/config/logging enabled
- `mock`: no model, useful for validating LangGraph/runtime behavior

EdgeVisor dynamic support:

- The EdgeVisor backend sets `DLLAMA_PLAN_CTRL_SOCKET`.
- It starts root inference with `--enable-plan-barrier`.
- During the configured generation it sends `set_plan` over UDS.
- The trace records UDS ping, set_plan response, status polling, and whether
  EdgeVisor logs contain plan emit/apply markers.

Example:

```bash
/home/byh/B01/agent_langgraph_venv/bin/python -m agent_bench.run_episode --backend mock

/home/byh/B01/agent_langgraph_venv/bin/python -m agent_bench.run_episode \
  --backend prima \
  --cuda-visible 0,1,2

/home/byh/B01/agent_langgraph_venv/bin/python -m agent_bench.run_episode \
  --backend edgevisor \
  --cuda-visible 0,1,2 \
  --edge-worker-gpus 1 \
  --edge-ratios 1:1
```

All outputs are written under `/home/byh/B01/agent_bench_results` by default.

## Agentic Ablation Harness

The ablation workload is intentionally an agent episode, not a single prompt
benchmark. The default episode is:

```text
agent_bench/episodes/agentic_ablation_episode.json
```

It forces multiple LLM generations, tool calls, observations, and replanning.
The `edgevisor_dynamic_plan` section injects a migration during one generation
inside the episode. `scripts/run_agentic_ablation_suite.sh` copies this episode
per run and tags the injected fluctuation as `compute`, `network`, or `mixed`.

Single variant example:

```bash
/home/byh/B01/agent_langgraph_venv/bin/python -m agent_bench.run_loop_episode \
  --backend edgevisor_ablation \
  --episode agent_bench/episodes/agentic_ablation_episode.json \
  --out-root /home/byh/B01/agentic_ablation_results/manual \
  --cuda-visible 0,1,2 \
  --edge-worker-gpus 1,2 \
  --edge-ratios 1:1:1 \
  --edge-steps 256 \
  --shadow-kv-mode enabled \
  --pointer-swizzling-mode operator_rebuild \
  --jit-mode enabled \
  --vg-mode enabled \
  --experiment-id pointer_operator_rebuild_manual
```

Every run writes `trace.json`. The top-level `agent_metrics` contains:

- `episode_id`, `episode_length`, `generation_id`, `tool_call_id`, `stage_id`
- `episode_completion_time`
- `episode_delay_over_baseline`
- `cumulative_stall_time`
- `affected_generation_latency`
- `recovery_latency`
- `max_token_stall`
- `p99_tpot`
- `task_success`

Each `llm_generation` event also carries the generation command, runtime
metrics, dynamic migration events, and parsed EdgeVisor ablation JSONL records
when `--ablation-log-path` is enabled.

Suite execution:

```bash
SMOKE=1 bash scripts/run_agentic_ablation_suite.sh
bash scripts/run_agentic_ablation_suite.sh
```

The suite refuses to run unless `CUDA_VISIBLE_DEVICES=0,1,2`, so GPU3 remains
untouched. It writes `manifest.jsonl`, per-run traces/logs, and summary files
generated by `scripts/summarize_ablation_results.py`.

For `edgevisor_ablation`, the Python backend also applies mode-specific runtime
policy before launching `dllama`:

- `shadow_kv_mode=enabled` keeps `--kv-redundancy 2`.
- `shadow_kv_mode=disabled_transfer` and `disabled_recompute` launch with
  `--kv-redundancy 0 --enable-kv-redundancy-during-migration 0`, so the run
  exercises the no-readiness path and records transfer/recompute fallback
  metadata or reject reasons in JSONL.
- `jit_mode=static` rewrites the UDS move list to a fixed policy.
- `jit_mode=greedy` rewrites the UDS move list to a resource-only greedy
  policy that ignores enactment/readiness cost.
- `vg_mode=flat`, `random`, `pure_pp`, and `no_elastic_vg` produce legal
  effective worker/ratio mappings and attach before/after VG mapping metadata
  to the UDS command and trace.
